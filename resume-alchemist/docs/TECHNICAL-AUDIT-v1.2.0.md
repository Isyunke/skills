## 一、架构层面：设计优秀，但实现有断层

系统架构设计是**真正出色的**——模块化子 skill、共享 schema、三条原则的工程强制、盲评估 sub-agent 的 context 隔离设计，这些都体现了成熟的系统思维。但存在几处"设计到实现的断层"：

### 1.1 模板渲染引擎缺失 🔴 严重

**位置**：`skills/resume-build/SKILL.md` Phase 4

**问题**：
- 设计文档描述了用 Jinja2 渲染模板（`from jinja2 import Template`）
- 但模板文件（`templates/official/*.html`）使用的是 Handlebars 语法（`{{#each}}`、`{{#if}}`）
- Python 端**没有渲染引擎代码**，`tools/` 目录下无渲染脚本
- resume-build 实际依赖 Claude 的上下文理解来填充模板，而非程序化渲染

**影响**：
- 模板变量名不一致时会静默失败
- 没有 fallback 逻辑
- 无法保证渲染结果的结构正确性
- 不同次生成的简历格式可能不一致

**修复方案**：

```
方案 A（推荐）：写一个 Python 渲染脚本
  - tools/template_renderer.py
  - 将模板语法统一为 Jinja2
  - 提供 CLI：python template_renderer.py --template tech-standard.html --data profile.json --output resume.html

方案 B：明确设计决策
  - 在 SKILL.md 中声明"模板是参考布局，实际由 Claude 直接生成 HTML"
  - 移除 Jinja2 依赖
  - 模板降级为"布局参考文档"而非"渲染模板"
```

**优先级**：🔴 P0 — 直接影响简历生成的核心流程

### 1.2 Hook 强制层只警告不阻止 🔴 严重

**位置**：`hooks/truth-verification.sh`、`hooks/evidence-chain.sh`

**问题**：
- 两个 hook 脚本最终都是 `exit 0`（只警告不阻止）
- 三条原则被定义为"不可妥协"，但在工程层面实际上是**可妥协的**
- 依赖 `jq` 命令，Windows 用户大概率没有安装
- 检查逻辑非常简单（grep 关键词），容易被绕过

**影响**：
- 用户可以生成没有证据链的简历
- "精通"级别的技能声明可以没有项目支撑
- 三条原则的核心差异化被削弱

**修复方案**：

```bash
# 阶段 1：增强检查逻辑
# - 不再只 grep "精通"，而是解析 HTML 结构
# - 检查每个技能标签是否有对应的项目描述
# - 检查量化数据是否存在

# 阶段 2：真正阻止
# - 在证据链不完整时 exit 1
# - 输出明确的修复指引

# 阶段 3：移除 jq 依赖
# - 用纯 bash 或 Python 重写解析逻辑
# - 确保 Windows 兼容性
```

**优先级**：🔴 P0 — 三条原则是核心差异化，必须工程强制

### 1.3 工具链脆弱 🟡 中等

#### keyword_matcher.py

**问题**：
- 关键词列表是硬编码的，只有约 40 个技术关键词
- 非技术岗（产品、运营、设计、市场）基本无效
- 没有行业特异性（"Python"在量化交易和 Web 开发的权重不同）
- 与 resume-jd 的语义匹配是**两套独立逻辑**，可能矛盾

**修复方向**：
- 支持用户自定义关键词库
- 引入行业关键词包（`keywords/tech.json`、`keywords/product.json`）
- 与 resume-jd 的分析结果整合，而非独立运行

#### resume_parser.py

**问题**：
- 正则在复杂简历格式上会失败（多栏排版、非标准 Markdown、图片中的文字）
- 姓名提取逻辑过于简单（取第一行短文本）
- 技能提取依赖预定义列表，无法发现列表外的技能
- 工作经历提取对中文公司名的正则覆盖不全

**修复方向**：
- 增加对 PDF 多栏排版的处理（pdfplumber 的 layout 参数）
- 引入 NLP 命名实体识别替代纯正则
- 支持用户手动修正解析结果

#### html_to_pdf.py

**问题**：
- weasyprint 在 Windows 上安装需要 GTK 等系统依赖，安装失败率高
- playwright 需要额外下载 ~100MB 的 chromium
- 没有检测当前环境的预检逻辑
- 错误信息不够友好

**修复方向**：
- 增加环境预检：`python html_to_pdf.py --check`
- 提供安装指引链接
- 考虑支持在线 PDF 生成 API 作为第三备选

### 1.4 缺少端到端集成测试 🟡 中等

**问题**：
- 只有 `test_resume_parser.py`、`test_keyword_matcher.py`、`test_html_to_pdf.py` 三个单元测试
- 没有子 skill 之间的集成测试
- 没有模拟完整用户流程的 E2E 测试
- 测试覆盖率极低

**修复方向**：
- 创建 `tests/integration/` 目录
- 编写完整的用户旅程测试：init → mine → jd → build → export
- 使用 fixture 数据（示例 profile、示例 JD）

### 1.5 状态迁移机制未实现 🟡 中等

**位置**：`migrations/registry.md`

**问题**：
- registry.md 定义了迁移规范，但没有实际的迁移脚本
- 当 schema_version 变化时，旧的 `.resume-state.json` 无法自动升级
- `state-schema.md` 提到"绝不让 skill 静默兼容旧版 schema"，但没有迁移工具

**修复方向**：
- 创建 `tools/migrate.py`
- 实现 1.0 → 1.1 的迁移逻辑
- 在 resume-init 中加入版本检测和自动迁移

## 二、功能缺口：需要真实测试才知道方向的 6 个核心问题

这些不是靠看代码能发现的，必须**找真实求职者跑完整流程**才能验证。

### 问题 1：STAR 深挖的真实效果如何？

**设计假设**：用户能通过 STAR 法则被引导出量化成果。

**现实可能**：
- 大多数人**真的不记得具体数字**——"性能提升了多少？""大概 50% 吧？"
- 有些岗位（如运营、行政、客服）**天然难以量化**
- 用户被追问时会感到**被审问**而非被引导
- 连续追问 3 次以上，用户可能产生抵触情绪

**需要测试**：
- 引导 3 个不同背景的人走完 resume-mine 流程
- 观察：他们能否产出量化数据？追问几次后会不会烦躁？
- 最终产出的 project.md 质量如何？
- 对比：有引导 vs 无引导，产出质量差异有多大？

**测试指标**：
| 指标 | 目标值 | 说明 |
|---|---|---|
| 完成率 | ≥ 80% | 用户愿意走完 STAR 四步 |
| 量化率 | ≥ 60% | 产出的成果描述包含数字 |
| 用户满意度 | ≥ 4/5 | 不觉得被审问 |
| 平均耗时 | ≤ 15 min/项目 | 每个项目的深挖时间 |

### 问题 2：JD 分析的匹配度算法是否准确？

**设计假设**：Claude 的语义理解能准确判断技能匹配度。

**现实可能**：
- 同一个 JD 分析两次可能结果不同（LLM 的非确定性）
- keyword_matcher 的硬编码关键词与 JD 分析的语义匹配是两套独立逻辑
- 没有行业特异性——"Python 精通"在量化交易和 Web 开发含义不同
- 匹配度百分比的计算方式不透明

**需要测试**：
- 对同一个 JD 分析 3 次，对比结果一致性
- 让真实招聘方评分匹配度，与系统评分对比
- 测试不同行业的 JD（技术/产品/运营/设计）

### 问题 3：简历导入的"马赛克项目"判定是否可靠？

**设计假设**：通过对话考察能区分"真实参与"和"参与但不熟"。

**现实可能**：
- 信心评估完全依赖 Claude 的主观判断，没有标准化
- 没有标准化的考察问题库
- 没有可复现的评分标准
- 用户没有申诉机制——被判定为"马赛克"可能感到被冒犯

**需要测试**：
- 让一个人导入包含 3 个真实项目和 2 个"参与但不熟"的项目
- 观察系统能否准确区分
- 观察用户对"马赛克项目"判定的情绪反应

### 问题 4：PDF 导出在真实环境能否跑通？

**设计假设**：用户能顺利安装 weasyprint 或 playwright。

**现实可能**：
- weasyprint 在 Windows 上需要 GTK 等系统依赖，安装失败率极高
- playwright 需要下载 ~100MB 的 chromium
- 非技术用户在这一步直接卡死
- 没有环境预检和友好的错误提示

**需要测试**：
- 在 3 台干净的电脑（Windows/Mac/Linux）上从零安装 → 导出 PDF
- 记录所有报错和耗时
- 测试不同简历模板的 PDF 渲染效果

### 问题 5：本地化的"文化适配"质量如何？

**设计假设**：Claude 能根据目标市场文化正确改写简历。

**现实可能**：
- 企业文化特征库只有 9 家公司，且描述非常泛化
- 英文简历改写质量取决于 Claude 的英文写作能力，无法保证一致性
- 没有英语母语人士的 review 机制
- "改写"和"翻译"的边界模糊

**需要测试**：
- 拿一份中文简历转换为英文版
- 给英语母语的招聘方 review
- 对比：改写版 vs 直译版，哪个更专业？

### 问题 6：多轮对话的上下文管理

**设计假设**：Claude Code 的上下文窗口足够处理所有文件。

**现实可能**：
- 当用户积累了 5 个项目 + 3 个 JD + 2 份简历 + 面试指导后，resume-build 需要读取大量文件
- 可能超出上下文窗口
- 可能遗漏重要信息
- 生成的简历可能与早期项目脱节

**需要测试**：
- 模拟重度使用场景：10 个项目 + 5 个 JD + 3 份简历
- 观察 resume-build 是否能正确引用所有项目
- 测试上下文溢出时的降级行为

## 三、求职用户的真实痛点 vs 系统覆盖度

当前求职市场（尤其中国市场）的核心痛点分析：

| # | 用户痛点 | 当前覆盖 | 缺什么 | 优先级 |
|---|---|---|---|---|
| 1 | "一份简历投百家公司，石沉大海" | ✅ 针对性原则 + JD 定制 | 批量生成能力 | P0 |
| 2 | "不知道自己的经历有什么价值" | ✅ STAR 深挖 | 需要真实验证深挖效果 | — |
| 3 | "简历被 ATS 系统过滤" | ⚠️ 关键词匹配有但太弱 | ATS 友好度检测 | P0 |
| 4 | "不会写自我评价/项目描述" | ⚠️ 有引导但无自动生成 | 一键生成 + 多风格选择 | P1 |
| 5 | "投了 100 家没有反馈，不知道问题在哪" | ❌ 完全缺失 | **投递跟踪 + 数据分析** | P0 |
| 6 | "面试问到简历上的东西答不上来" | ✅ 面试准备 + 马赛克检测 | 交互式模拟面试 | P1 |
| 7 | "不知道薪资怎么谈" | ❌ 完全缺失 | **薪资指导 + 市场数据** | P1 |
| 8 | "英文简历不知道怎么写" | ✅ 本地化功能 | 需要真实验证质量 | — |
| 9 | "转行不知道怎么包装" | ⚠️ 有学习指导但弱 | 转行专用策略 | P2 |
| 10 | "多个 offer 不知道怎么选" | ❌ 完全缺失 | **Offer 对比工具** | P2 |

## 四、建议加入的功能（按优先级排序）

### 🔴 P0：必须做，直接影响核心价值

#### 4.1 投递跟踪系统 (`resume-track`)

**痛点**：投了 100 家不知道哪些有回音，不知道问题在哪。

**功能设计**：

```
触发词："记录投递" / "投递状态" / "求职数据"

数据结构：tracking/applications.json
{
  "applications": [
    {
      "id": "app-001",
      "company": "XX 公司",
      "position": "后端工程师",
      "jd_id": "jd-001",
      "resume_version": "resume-v2.html",
      "applied_at": "2026-06-14",
      "status": "applied|read|interview|offer|rejected",
      "status_history": [...],
      "feedback": "",
      "notes": ""
    }
  ]
}

分析维度：
- 哪个版本的简历通过率高？
- 哪个行业的回复率高？
- 投递到回复的平均时间
- 拒绝原因分析（如果有反馈）
- 最佳投递时间段
```

**集成**：
- 上游：`resume-build` 生成简历时记录版本
- 下游：`resume-status` 显示投递数据
- 下游：`resume-optimize` 根据拒绝反馈优化简历

**价值**：这是**求职闭环**的关键一环。当前系统从"深挖经历"到"生成简历"是完整的，但缺少"投递 → 反馈 → 优化"的闭环。

#### 4.2 ATS 友好度检测 (`resume-ats`)

**痛点**：简历被机器过滤，HR 人看不到。

**功能设计**：

```
触发词："检查 ATS" / "ATS 友好度"

检测维度：
1. 格式检测
   - 是否使用表格（ATS 不友好）
   - 是否使用图片/图标（ATS 无法解析）
   - 是否使用非标准字体
   - 文件格式（PDF vs DOCX 的兼容性）

2. 关键词密度
   - JD 关键词在简历中的出现次数
   - 关键词分布是否均匀（不能全堆在技能清单）
   - 是否使用了 JD 的原词（同义词 ATS 可能识别不了）

3. 结构检测
   - 是否有标准的 section 标题（工作经历、教育背景）
   - 日期格式是否标准
   - 联系方式是否在显眼位置

输出：ATS 友好度评分（0-100）+ 具体修复建议
```

**集成**：
- 在 `resume-build` 后自动运行
- 在 `resume-optimize` 中作为优化维度之一

#### 4.3 批量 JD 管理 (`resume-batch`)

**痛点**：投多家公司时，每个 JD 都要手动分析 + 手动生成简历。

**功能设计**：

```
触发词："批量分析" / "批量生成" / "海投模式"

流程：
1. 用户粘贴多个 JD（或从文件导入）
2. 系统批量分析，生成匹配度排序
3. 用户选择要投的 JD
4. 一键批量生成定制简历
5. 生成投递优先级推荐（Top 5 最值得投的）

输出：
- 匹配度排序表
- 每个 JD 的关键差异点
- 批量生成的简历文件
- 投递建议（先投哪个、后投哪个）
```

### 🟡 P1：应该做，显著提升用户体验

#### 4.4 薪资指导 (`resume-salary`)

**痛点**：不知道自己值多少钱，面试被压价。

**功能设计**：

```
触发词："薪资参考" / "我值多少" / "谈薪"

功能：
- 根据经验、技能、城市给出薪资范围参考
- 薪资谈判话术（如何回应"你的期望薪资是多少"）
- 不同公司的薪资结构分析（base/股票/签字费/年终）
- Offer 对比表（多维度：薪资、成长、work-life balance）

数据来源：
- 公开的薪资数据（如脉脉、Glassdoor）
- 用户提供的市场信息
- 基于 JD 的薪资推断
```

#### 4.5 交互式模拟面试 (`resume-mock`)

**痛点**：面试准备是静态文档，没有真实模拟感。

**功能设计**：

```
触发词："模拟面试" / "面试模拟" / "Mock Interview"

流程：
1. 系统读取简历 + JD 分析
2. 模拟面试官提问（基于 JD + 简历）
3. 用户回答
4. 系统给出反馈：
   - 回答完整度（STAR 结构是否完整）
   - 量化程度（是否有数据支撑）
   - 时间控制（是否太长或太短）
   - 追问模拟（面试官会根据回答深入追问）
5. 生成面试复盘报告

与 resume-interview 的区别：
- resume-interview = 静态面试题库（文档）
- resume-mock = 交互式模拟（对话）
```

```

   - 安装 Claude Code
   - 安装 resume-alchemist
   - 确认依赖可用

   - 运行 /resume-init

   - 确认目录结构创建正确

   - 粘贴一个真实 JD
   - 运行 /resume-jd
```
